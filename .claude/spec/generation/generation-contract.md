# 생성 계약서

CS 학습 콘텐츠(lesson·문제·선지·정답·해설) 생성의 단일 규범이다. 원칙마다 **규칙 → 좋은 예 → 피하라(안티패턴)** 순으로 둔다. 좋은 예는 형태를 그대로 베끼지 말고 본보기로 삼아 변형하라. "피하라"는 그대로 만들지 마라. INV/EXP/AP 코드는 검수·회고에서 신호로 쓰이니 보존한다.

## §0 기본 구성

- 스킬 1회 호출당 lesson 1개 = OBJECTIVE 4문제 + SUBJECTIVE 2문제 = 총 6문제.
- 객관식: 선지 4개(오답 3 + 정답 1, **is_answer**로 구분). 선지 품질은 §4.
- 주관식: 정답 1개(짧은 단답 — 용어·값·짧은 구). **~를 서술하시오**식 장문 답 금지. 정답·본문 규칙은 §5.

## §1 본문은 풀이에 꼭 필요해야 한다 (INV-1)

**규칙.** 본문은 장식이 아니라 정답을 결정하는 자료(추적할 코드·계산할 값·판별할 사례)다. **본문 삭제 테스트** — 타겟층(CS 배경 취준생·학부생) 기준으로 본문을 지워도 정답이 하나로 정해지면 본문이 일을 안 한 것이다. 사전지식만으로 풀리거나, 정답 선지가 본문 문장의 재진술이면 실패다.

**좋은 예 (개념 판별 객관식).**
- 발문: **아래 커널 구조에 대한 설명으로 옳은 것은?**
- 본문: **마이크로커널은 스케줄링·IPC 등 최소한의 기능만 커널 공간에 두고, 파일 시스템·드라이버 등은 사용자 공간의 독립 서비스로 분리한 구조이다.**

| 선지 | is_answer | 해설 |
|---|---|---|
| 서비스 하나가 죽으면 시스템 전체가 곧바로 패닉에 빠진다. | false | 모놀리식 약점을 갖다 붙인 오개념. 서비스가 분리돼 한 서비스 장애에 오히려 강하다. |
| 사용자 공간 서비스 간 요청이 IPC를 거쳐 모놀리식보다 통신 오버헤드가 크다. | true | 기능 분리의 대가 — 최소 커널의 핵심 트레이드오프. |
| 드라이버가 모두 커널 모드에서 실행돼 모드 전환이 없다. | false | 모놀리식 동작을 갖다 붙인 오개념. 드라이버가 사용자 공간이라 모드 전환이 있다. |
| 커널 코드가 많아 이식·유지보수가 어렵다. | false | 모놀리식 단점을 거꾸로 붙인 오개념. 커널이 작아 오히려 유리. |

본문을 지우면 정답(IPC 오버헤드)과 오답(이식성)이 둘 다 "맞는 말"이 돼 정답이 안 갈린다 → 본문이 정답을 정한다.

**피하라.**
- **공허한 본문 (AP-01).** 본문이 "커널은 OS의 핵심 기능을 담당한다" 같은 한 줄 주제 딱지뿐 → 지워도 풀이가 같다(+발문도 "관련하여 옳은 것은?"처럼 막연). 본문을 개념 설명 문장으로 끌어올리고 발문은 큰 갈래만 가리켜라(§2).
- **재진술 (AP-09).** 정답 선지가 본문 문장을 어순만 바꿔 되풀이(객관식)하거나, 본문이 정답 용어의 사전식 정의·기능을 그대로 서술(주관식 → §5). 본문 삭제 테스트는 통과해도 정답이 본문에 누설돼 변별력이 죽는다. 정답을 본문에 **없는** 다른 특징(동작 결과·비교·세대 차이·다른 속성)으로 옮겨라. 표가 본문이면 표 한 행을 복사하지 말고 표 값에서 **따라 나오는 결과·함의**(예: "타임 퀀텀이 짧을수록 문맥 전환이 잦아 오버헤드가 커진다")로 옮겨라.

## §2 발문은 무엇을 묻는지 분명해야 한다 (INV-2)

**규칙.** 발문은 본문과 함께 읽어 무엇을 묻는지 분명해야 한다.
- **개념 판별 문제:** 발문에 큰 갈래(예: **커널 구조**)까지만 적고 구체 개념(**마이크로커널**)은 본문에서 밝힌다. 발문이 개념 이름을 직접 부르면 본문이 할 일이 없어진다.
- **자료 해석·계산 문제:** 발문이 묻는 대상(값·결과·자료)을 직접 가리켜도 된다(예: **실행 후 v의 최종 값은?**).

- (O) 아래 커널 구조에 대한 설명으로 옳은 것은? — 큰 갈래만
- (X) 마이크로커널에 대해 옳은 것은? — 개념 직접 호명, 본문이 군더더기
- (X) 제시된 내용과 관련하여 옳은 것은? — 무엇을 묻는지 막연

(좋은 예는 §1 참고.)

## §3 정답은 하나로 정해져야 한다 (INV-3, INV-6)

**규칙.**
- 본문+발문만으로 정답이 유일해야 한다. 합당한 해석으로 답이 갈리면 안 된다.
- 객관식: 정답 1개, 나머지 3개는 어떤 합당한 해석으로도 정답이 될 수 없다.
- 주관식: 허용 정답을 빠짐없이 모은다(§5).
- **풀이에 필요한 전제가 모두 본문·발문 안에** 있어야 한다(INV-6). 트랜잭션이면 초기값, 페이징이면 페이지 크기+주소 비트 수.

**좋은 예 (계산 전제 객관식).**
- 발문: **아래 조건에서 단일 단계 페이지 테이블의 엔트리 개수는?**
- 본문: **가상 주소 공간은 32비트이고, 페이지 크기는 4KB이다.**

| 선지 | is_answer | 해설 |
|---|---|---|
| 4,096 | false | 오프셋 비트 수(12)를 엔트리 수로 혼동. 오프셋은 페이지 내부 위치일 뿐. |
| 1,048,576 | true | 페이지 번호 비트 = 32 − 12 = 20 → 2²⁰. 4KB=2¹²라 오프셋 12비트. |
| 4,194,304 | false | 페이지 크기를 1KB로 착각해 오프셋 10비트로 본 것(2²²). |
| 4,294,967,296 | false | 페이지 크기로 나누지 않고 주소 공간 전체(2³²)를 엔트리 수로 본 것. |

전제(주소 비트+페이지 크기)가 다 있어 정답이 하나로 떨어진다.

**피하라.**
- **전제 누락 (AP-04).** 페이지 크기 없이 엔트리 수를 물으면 답이 안 정해진다 → 계산 전제를 빠짐없이 적어라.
- **해석 분기 (AP-06).** "메모리를 효율적으로 쓰는 기법" + 페이징/세그멘테이션/가상메모리/압축 → 여러 개가 답이 된다. 본문에 정답을 유일하게 만드는 판별 특징(예: "고정 크기 블록으로 외부 단편화 제거" → 페이징만)을 넣고, "가장 ~한"류 주관적 비교 발문을 피하라.

## §4 선지 품질 (INV-4, INV-5)

**규칙.**
- 네 선지는 같은 기준 위에(값이면 값, 특징이면 특징) 있어야 하고, **발문이 묻는 단위와 선지 단위가 같아야** 한다(INV-5). "v의 값은?"이면 선지 모두 값.
- 오답 3개는 학습자가 흔히 갖는 오개념에서. 너무 쉽게 지워지면 안 된다.
- 정답만 유독 길거나 자세하면 내용을 몰라도 새어 나간다 — 네 선지의 길이·구체성을 맞춰라.
- 개념 판별 문제: 선지에 개념 이름(주어)을 쓰지 말고 특징만. 본문에 적힌 내용을 되풀이 말고 본문에 없는 특징을 다뤄라(§1 AP-09).
- **부정 발문(옳지 않은 것은?):** 정답은 거짓 진술 1개, 오답 3개는 모두 흠 없는 참인 진술. **is_answer=true**는 "옳은 선지"가 아니라 "고르라고 한 선지(=거짓)"다. 해설은 정답에선 어디가 거짓인지, 오답에선 왜 참인지 짚는다(§6).

**좋은 예 (자료 해석 객관식).**
- 발문: **아래 트랜잭션을 실행한 뒤 컬럼 v의 최종 값은?**
- 본문(코드):

```sql
-- 초기 상태: v = 10
UPDATE t SET v = 20 WHERE id = 1;
SAVEPOINT sp1;
UPDATE t SET v = 30 WHERE id = 1;
ROLLBACK TO sp1;
COMMIT;
```

| 선지 | is_answer | 해설 |
|---|---|---|
| 10 | false | ROLLBACK TO sp1을 전체 롤백으로 오해. sp1 이전 UPDATE(v=20)는 남는다. |
| 20 | true | ROLLBACK TO sp1은 sp1 이후 변경(v=30)만 취소. v=20이 남고 COMMIT으로 확정. |
| 30 | false | ROLLBACK TO를 무시하고 마지막 UPDATE가 반영된다고 오해. |
| null | false | 롤백이 행/값을 지운다고 오해. ROLLBACK TO는 행을 지우지 않는다. |

선지가 모두 **값** 한 단위(INV-5), 오답마다 다른 트랜잭션 오개념을 담았다.

**좋은 예 (표 해석 · 부정 발문).**
- 발문: **위 비교표를 바탕으로 옳지 않은 것은?**
- 본문(표): FCFS=비선점·도착순 / SJF=비선점·짧은작업우선 / RR=선점·타임퀀텀순환.
- 정답(거짓): **RR은 비선점이라 한 작업이 끝날 때까지 CPU를 놓지 않는다** — 표의 RR=선점에 정면으로 걸린다. 오답 3개는 표에서 도출되는 참인 진술(FCFS 호위 효과 등).

**피하라.**
- **쉽게 지워지는 오답 (AP-02).** "정렬을 안 한다"·"항상 O(n!)"·"문자열에만" 같은 터무니없는 오답 → 개념 몰라도 소거법으로 풀림. 오답을 실제 헷갈리는 오개념(최악 O(n²) 혼동, 안정성·메모리 오부여)으로 바꿔라.
- **단위 혼합 (AP-03).** 값을 묻는데 선지에 "20"·"30"과 "롤백됨"·"오류로 중단"이 섞임 → 형태가 정답 단서. 한 단위로 통일하라.
- **정답만 긴 선지 (AP-07).** 정답에만 조건·예외를 길게 → "가장 신중해 보이는 것" 찍기로 새어 나감. 길이·구체성을 맞춰라.

## §5 주관식 본문·정답 (INV-1·INV-3의 주관식 적용)

**규칙.**
- 정답은 짧은 단답. content에 허용 표현을 쉼표로 빠짐없이(한글 표기 변형 + 영문 + 약어 + 동의어). 대소문자 무시·문자열 일치 채점이라 빠지면 맞고도 오답 처리된다.
- **본문이 정답의 정의·기능을 그대로 서술하지 마라(재진술, §1 AP-09).** 정의뿐 아니라 정답이 가리키는 기능·역할을 풀어 쓴 것("어떤 방식으로 서버와 통신할지 정한다", "데이터를 잘라 머리말을 붙여 보낸다")도 사전식 서술이다. 본문은 **상황·증상·사례·효과·수치**로 개념을 간접적으로 가리켜, 학습자가 "이 현상의 이름이 뭐더라"를 떠올려 풀게 하라.
- **상황·장면이 자연히 안 떠오르는 정적 개념(자료구조 종류·SQL 절·제약조건 등)** 은 다음 순으로 본문을 짜라. 어느 것도 정답의 정의·기능을 문장으로 쓰지 않는다.
  1) **효과·결과 수치** — 쓰기 전/후 동작·성능 차이 (예: 1.8초→6ms)
  2) **오용·실패 증상** — 안 쓰거나 잘못 쓰면 생기는 문제
  3) **구체 인스턴스** — 정의가 아니라 그 개념에 해당하는 구체 사례 하나
  4) **비교 장면** — 옆 개념과 결과가 갈리는 한 장면

**좋은 예 (상황 → 용어).**
- 발문: **아래 상황에서 발생한 문제를 가리키는 용어는?**
- 본문: **두 스레드 A·B가 공유 변수 counter를 각각 1씩 증가시킨다. 둘 다 counter를 읽은 뒤 1을 더해 다시 쓰는데, 실행이 엇갈리면 두 번 증가시켰는데도 최종값이 1만 늘어난다.**
- 정답: **경쟁 상태,경쟁 조건,race condition,레이스 컨디션**
- 해설: 동기화되지 않아 인터리빙에 따라 결과가 달라지는 게 경쟁 상태. 교착 상태(deadlock)·임계 구역(critical section)과 헷갈리지 않게(§6 EXP-4).

**좋은 예 (정적 개념 → 효과·수치).**
- 발문: **아래 상황에서 조회 성능을 끌어올리려고 추가한 데이터베이스 객체의 이름은?**
- 본문: **회원 테이블 200만 행에서 `email = ?` 조회가 평균 1.8초 걸렸다. 특정 컬럼을 키로 하는 별도 자료를 만들자 같은 조회가 6ms로 줄었고, 대신 INSERT·UPDATE는 다소 느려졌다.**
- 정답: **인덱스,index**
- 해설: 조회를 빠르게 하려고 특정 컬럼을 정렬해 따로 관리하는 보조 자료가 인덱스. 조회는 빨라지지만 쓰기마다 갱신 비용이 붙는 트레이드오프가 핵심. 풀 스캔과 구분. — 정의를 한 번도 말하지 않고 효과(1.8초→6ms)·비용으로만 가리킨 본보기.

**피하라.**
- **정답 목록 누락 (AP-08).** 정답을 "슬로 스타트"만 두면 slow start·슬로우 스타트·느린 시작을 쓴 학습자가 오답 처리된다. 한글 변형+영문+약어+동의어를 다 나열하라.

## §6 해설 (EXP-1~4)

**규칙.** 선지마다(**option.explanation**), 주관식 정답마다(**answer.explanation**) 해설을 단다.
- **EXP-1.** 결론 되풀이 금지("②가 옳으므로 정답"). 핵심 개념으로 근거를 대라. (O) "최소 기능만 커널에 두고 나머지를 사용자 공간으로 분리하므로 호출이 IPC를 거쳐 통신 오버헤드가 생긴다."
- **EXP-2.** 오답 해설엔 그 선지를 고르게 만든 오개념을 짚어라(§4와 짝). 부정 발문의 오답(참인 진술)은 "참이다"로 끝내지 말고 왜 참인지 근거를.
- **EXP-3.** 해설만으로 배울 거리가 있어야. 자료 해석 문제는 해설이 본문을 가리켜도 좋다("본문의 ROLLBACK TO sp1 때문에…").
- **EXP-4.** 주관식 해설엔 정답 근거 + 헷갈리는 옆 개념과의 경계를 함께(예: IP는 논리 주소라 네트워크 계층(3), MAC으로 옆 노드에 전달하는 데이터링크 계층(2)과 구분).

**피하라.**
- **결론만 되풀이하는 해설 (AP-05).** "②가 옳으므로 정답"·"①은 틀렸다" → 배울 게 없다. 정답은 개념 근거, 오답은 그 오개념을 짚어라.

## §7 레슨 단위 구성

- 개념노트의 핵심 개념을 6문제에 고르게 분배. 특정 개념 편중 금지.
- 기존 문제 목록을 반드시 확인. 개념 반복은 복습 효과로 권장하되, **같은 개념 + 같은 본문 유형 + 같은 정답 포인트**가 모두 겹치면 중복(금지). 셋 중 하나만 달라도 허용.
- 난이도: 타겟층(CS 배경 취준생·학부생) 기준 개념 이해 + 약간의 응용.
- 6문제 본문이 한 형태로 쏠리지 마라 — 최소 두 종류 이상의 역할(개념 설명 문장 · 코드/로그 · 표/비교 자료 · 상황·사례). 모두 "개념 설명 문장" 한 가지면 안 된다.
- 객관식 정답 선지 위치를 문항마다 분산(한 번호에 몰리지 않게).
- **위 모두 옳다 / 정답 없음 / ①과 ③** 같은 메타 선지 금지 — 각 선지는 그 자체로 참·거짓을 판단할 수 있어야 한다.
